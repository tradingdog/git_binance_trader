# AI自治系统落地清单（超详细执行版）

## 0. 文档目的与使用方式
- 目的：把“可控自治”框架从概念变为可逐条执行、可验收、可回滚的工程计划。
- 使用方式：
  - 每完成一个条目即勾选，并在对应阶段记录证据（PR、报告、截图、日志路径）。
  - 每个阶段都要有“通过/阻断”结论，阻断后不得进入下一阶段。
  - 本清单默认仅针对 `ai_trader_project/`，不影响人类版本工程。

---

## 1. 总体目标（固定）

### 1.1 核心目标
- [ ] 人类只设目标与红线，AI 自动完成研究、改码、验证、灰度、复盘。
- [x] 人类始终保留最高权限（暂停、平仓、停止、回滚、冻结自治）。
- [x] 全程不突破模拟盘与风控红线。

### 1.2 不可突破硬红线
- [x] 全程最大回撤 <= 15%。
- [x] 单日最大回撤 <= 5%。
- [x] 单笔亏损 <= 1%。
- [x] 禁止实盘链路（默认 `SIMULATION`）。
- [x] AI 无权修改硬红线（仅 Human Root 可改）。

### 1.3 模型策略（Gemini）
- [x] 稳定通道：默认 `Gemini 2.5 Pro`（生产稳定口径）。
- [x] 实验通道：`Gemini 3.1 Pro` 预览（仅挑战者/实验）。
- [x] 区域策略：实现 `global` 与区域回退逻辑（可观测失败原因）。
- [x] 服务账号检查：IAM、配额、账单、区域可用性自动探针。

## 4. 机器可读目标与评分体系

### 4.1 三层目标结构
- [x] 硬约束（不可突破）写入独立配置并上锁。
- [x] 软目标（收益/稳定性/手续费）可配置。
- [x] 评分函数 `J` 参数可调并可审计。

### 4.2 评分函数实现
- [x] 定义指标：`R_day`、`Sharpe`、`MDD`、`FeeRatio`、`TurnoverPenalty`。
- [x] 实现公式：
  - $J = w_1 \cdot R_{day} + w_2 \cdot Sharpe - w_3 \cdot MDD - w_4 \cdot FeeRatio - w_5 \cdot TurnoverPenalty$
- [x] 若触发硬约束则直接判失败（短路机制）。
- [x] 评分过程全量写审计日志。

### 4.3 验收标准
- [x] 每次策略候选都有同一评分口径。
- [x] 指标计算可重复（同输入同输出）。
- [x] 评分失败路径可解释。

## 5. 权限体系与密钥治理（先做）

### 5.1 角色与权限拆分
- [ ] 研究/改码 AI：仅代码分支与 PR 权限。
- [x] 验证 AI：仅测试/回测执行权限。
- [x] 发布 AI：仅发布/回滚权限，不可改代码。
- [x] Human Root：冻结、回滚、撤销密钥、禁用策略类型。

### 5.2 凭据与命令安全
- [x] 所有密钥仅在 Secret 存放，禁止写入仓库。
- [x] 建立命令白名单（test/backtest/build/deploy allowed）。
- [x] 拒绝高危命令（系统破坏、越权操作）。
- [x] 增加密钥泄漏扫描（CI 阶段阻断）。

### 5.3 主干保护
- [x] 主分支禁止直推。
- [x] 必须通过：单测、回测、风控阈值检查、密钥扫描。
- [x] 高风险改动必须审批通过方可合并。
